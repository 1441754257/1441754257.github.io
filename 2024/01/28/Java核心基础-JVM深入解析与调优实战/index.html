二、垃圾回收机制

2.1 垃圾判断算法

引用计数法（已被淘汰）

  • 原理：对象被引用时计数+1，引用失效时-1
  • 缺点：无法解决循环引用问题

可达性分析法（JVM采用）

  • 原理：从GC Roots出发，搜索引用链
  • GC Roots：虚拟机栈引用、方法区静态属性、方法区常量、本地方法栈引用

2.2 垃圾回收算法

标记-清除算法

  • 标记：标记所有需要回收的对象
  • 清除：统一回收被标记对象
  • 缺点：效率低、产生内存碎片

复制算法（新生代采用）

  • 将内存分为两块，每次使用一块
  • 回收时将存活对象复制到另一块
  • 优点：效率高、无碎片
  • 缺点：内存利用率低

标记-整理算法（老年代采用）

  • 标记存活对象
  • 将存活对象整理到一端
  • 优点：无碎片、内存利用率高

2.3 垃圾收集器对比

收集器 区域 特点 适用场景
Serial 新生代 单线程、简单 客户端模式
ParNew 新生代 Serial多线程版 配合CMS使用
Parallel Scavenge 新生代 吞吐量优先 后台计算任务
CMS 老年代 并发收集、低停顿 互联网应用
G1 全区域 分区收集、可预测停顿 大内存应用
ZGC 全区域 染色指针、极低延迟 超大内存应用

五、调优案例

5.1 案例一：频繁Full GC

现象

  • Full GC频繁触发
  • 应用响应时间长
  • CPU使用率高

分析

解决方案

  1. 增大堆内存：-Xmx8g
  2. 调整新生代比例：-XX:NewRatio=1
  3. 使用G1收集器：-XX:+UseG1GC

5.2 案例二：内存泄漏

现象

  • 老年代持续增长
  • Full GC无法回收
  • 最终OOM

分析步骤

常见泄漏原因

  • 静态集合类持有对象引用
  • 未关闭的连接（数据库、IO）
  • ThreadLocal未清理
  • 监听器未注销

六、最佳实践

6.1 内存配置原则

  • Xms = Xmx（避免动态扩容开销）
  • 新生代占比：根据对象生命周期调整
  • 元空间：根据类加载量调整

6.2 GC选择建议

应用类型 推荐收集器 原因
批处理任务 Parallel 吞吐量优先
交互式应用 G1 低延迟
超大堆(>16G) ZGC 极低延迟
小堆(<4G) Parallel 简单高效

6.3 监控告警

  • GC频率监控
  • GC停顿时间监控
  • 堆内存使用率监控
  • 建立基线，异常告警

总结

JVM调优是一个系统工程，需要：

  1. 理解JVM内存模型和GC原理
  2. 掌握常用诊断工具
  3. 结合业务特点进行调优
  4. 持续监控和优化

💡 建议：在生产环境开启GC日志，定期分析，建立性能基线，及时发现和解决问题。
