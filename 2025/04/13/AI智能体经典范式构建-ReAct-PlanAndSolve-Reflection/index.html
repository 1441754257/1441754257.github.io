AI智能体经典范式构建-ReAct-PlanAndSolve-Reflection

本文基于 Datawhale《Hello-Agents》第四章内容，总结智能体（Agent）的三大经典构建范式：ReAct、Plan-and-Solve 和 Reflection，通过实战代码解析智能体如何连接大语言模型与外部世界，实现复杂任务的自动化执行。

一、智能体概述

1.1 什么是智能体

智能体（Agent）的核心能力在于将大语言模型的推理能力与外部世界联通。它能够：

  • 自主理解用户意图
  • 拆解复杂任务
  • 调用代码解释器、搜索引擎、API 等工具
  • 获取信息、执行操作、达成目标

1.2 智能体的能力边界

智能体并非万能，同样面临挑战：

  • 幻觉问题：LLM 本身可能产生错误信息
  • 推理循环：在复杂任务中可能陷入无限循环
  • 工具误用：对工具的错误使用导致任务失败

1.3 为什么需要范式

尽管 LangChain、LlamaIndex 等框架已经很成熟，但直接使用高度抽象的工具，不利于了解背后的设计机制。

亲手实现智能体范式，可以：

  1. 深入理解机制：知道工具调用、重试、死循环防护是如何工作的
  2. 培养系统设计能力：处理输出解析、格式规范等工程挑战
  3. 成为创造者而非使用者：当标准组件无法满足需求时，能深度定制甚至从零构建

三、ReAct 范式（Reasoning + Acting）

3.1 核心思想

ReAct 由 Shunyu Yao 于 2022 年提出，核心是 “思考-行动-观察” 循环：

  • Thought：分析当前情况、分解任务、制定下一步计划
  • Action：调用外部工具，如 Search['查询内容']
  • Observation：执行后的结果，如搜索结果摘要

3.2 数学形式化

在每个时间步 t，智能体根据历史轨迹生成思考和行动：

$$
(th_t, a_t) = \pi(q, (a_1, o_1), \ldots, (a_{t-1}, o_{t-1}))
$$

环境执行行动并返回观察：

$$
o_t = T(a_t)
$$

核心结论：推理使行动更具目的性，行动为推理提供事实依据。

3.3 工具的定义

一个良好的工具包含三个核心要素：

要素 说明
名称 (Name) 唯一标识符，如 Search
描述 (Description) 说明工具用途，是最关键的部分
执行逻辑 真正执行任务的函数

工具描述是整个机制的关键 —— LLM 依赖这段描述来判断何时使用哪个工具。

3.7 ReAct 的特点与局限性

主要特点：

  • 高可解释性：通过 Thought 链清晰展示决策过程
  • 动态纠错：根据 Observation 动态调整后续计划
  • 工具协同：LLM 负责推理，工具负责执行

固有局限：

  • 对 LLM 自身能力强依赖（格式遵循、逻辑推理）
  • 执行效率问题（每步都需调用 LLM）
  • 提示词脆弱性（微小变动可能破坏流程）
  • 可能陷入局部最优，缺乏全局规划

适用场景：需要外部知识、精确计算、与 API 交互的任务。

四、Plan-and-Solve 范式

4.1 核心思想

Plan-and-Solve 由 Lei Wang 于 2023 年提出，”三思而后行“：

先制定完整行动计划，再严格执行。

4.2 两阶段工作流

规划阶段：

$$
P = \pi_{\text{plan}}(q)
$$

执行阶段：

$$
s_i = \pi_{\text{solve}}(q, P, (s_1, \ldots, s_{i-1}))
$$

最终答案 = 最后一个步骤的执行结果 $s_n$。

4.5 Plan-and-Solve 的优势

特点 说明
结构性 清晰的计划使任务分解一目了然
稳定性 严格按照计划执行，不易偏离目标
全局视野 事先规划，避免被中间步骤带偏
可追溯 每个步骤的执行结果清晰可查

适用场景：多步数学应用题、结构化报告撰写、代码生成任务。

五、Reflection 范式（自我反思）

5.1 核心思想

Reflection 赋予智能体”反思“能力，通过自我批判和修正来优化结果。核心是”执行 → 反思 → 优化“循环：

5.2 迭代公式

$$
F_i = \pi_{\text{reflect}}(\text{Task}, O_i) \quad \text{（反思生成反馈）}
$$

$$
O_{i+1} = \pi_{\text{refine}}(\text{Task}, O_i, F_i) \quad \text{（优化生成新版本）}
$$

5.6 成本收益分析

主要成本：

  • 每轮迭代至少额外调用 2 次 LLM（反思 + 优化）
  • 任务总耗时显著延长
  • 提示工程复杂度上升

核心收益：

  • 解决方案质量从”合格”跃升到”优秀”
  • 发现并修复逻辑漏洞、事实错误
  • 构建短期”记忆”，积累优化经验

适用场景：对准确性、可靠性要求极高，且对实时性要求宽松的场景。

六、三种范式对比与选择策略

范式 核心策略 思考与行动关系 适用场景
ReAct 边想边做，动态调整 融合在每一步 探索性任务、外部信息获取
Plan-and-Solve 三思而后行 分离（先规划，后执行） 逻辑路径确定、内部推理密集
Reflection 执行-反思-优化 迭代循环 对质量要求极高的复杂任务

选择建议

  • 需要实时交互、工具调用 → ReAct
  • 任务结构清晰、多步骤推理 → Plan-and-Solve
  • 对结果质量和可靠性要求极高 → Reflection
  • 复杂场景可以组合使用：如 ReAct + Reflection

七、实战案例：素数求解的迭代优化

Reflection 机制驱动智能体将初版代码（O(n·√n)）优化为埃拉托斯特尼筛法（O(n log log n)）：

  1. 初始执行：生成简单的试除法实现
  2. 第一轮反思：指出时间复杂度瓶颈，建议筛法
  3. 第一轮优化：实现埃拉托斯特尼筛法
  4. 第二轮反思：肯定筛法效率，建议可考虑分段筛法
  5. 第二轮反思：判断”无需改进”，收敛终止

这展示了 Reflection 如何驱动解决方案的阶梯式质量提升。

八、调试技巧

ReAct 调试

  • 检查完整的提示词（含历史记录）
  • 分析 LLM 返回的原始输出
  • 在提示词中加入 Few-shot 示例引导格式遵循

Plan-and-Solve 调试

  • 验证规划阶段生成的计划结构是否合理
  • 检查执行阶段历史结果的传递是否正确
  • 可设计”动态重规划”机制应对执行失败

Reflection 调试

  • 使用不同能力的模型分别执行和反思
  • 设计更智能的终止条件（而非简单关键词匹配）
  • 对反思提示词进行角色设定调优
